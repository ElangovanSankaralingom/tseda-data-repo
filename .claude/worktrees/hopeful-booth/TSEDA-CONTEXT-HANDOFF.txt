## ENTRY STATUSES (6 total)

```
DRAFT → GENERATED → EDIT_REQUESTED → EDIT_GRANTED → (re-finalized)
                  → DELETE_REQUESTED → ARCHIVED
```

| Status | Meaning |
|--------|---------|
| DRAFT | Entry created, fields being filled |
| GENERATED | All mandatory data fields saved + PDF generated, timer running |
| EDIT_REQUESTED | User requested edit on finalized entry |
| DELETE_REQUESTED | User requested delete on finalized entry |
| EDIT_GRANTED | Admin approved edit request (1/2/3 days) |
| ARCHIVED | Entry hidden from user (admin-approved delete or auto-archive) |

## ENTRY LIFECYCLE

### Two-Stage Field Model
- **Stage 1 (data fields)**: programName, startDate, endDate, organisingBody, supportAmount, academicYear, etc. — required for Generate Entry. Changes make PDF stale.
- **Stage 2 (file uploads)**: permissionLetter, completionCertificate — required for Finalise Now only. Changes do NOT make PDF stale.

### Timer
- Non-streak entries: 3 days from committedAtISO (when entry first transitioned to GENERATED)
- Streak entries: endDate + 8 days
- Timer NEVER resets once started
- Timer expired + valid PDF → entry finalizes (read-only)
- Timer expired + NO valid PDF → auto-archive by cron

### Save Button Behavior
- "Save" and "Save & Close": enabled when form is dirty (any field changed). No mandatory field validation. Saves partial data.
- Cancel: always enabled
- Generate Entry: requires all Stage 1 mandatory DATA fields filled
- Finalise Now: requires PDF generated + not stale + all fields including uploads

### Generate Entry Button
- Creates PDF only (decoupled from status transition)
- Status transition DRAFT→GENERATED happens in engine.ts when commitDraft is called
- Visible when: no PDF yet OR PDF stale from Stage 1 edits
- Hidden when: fresh PDF exists
- After clicking: shows spinner → "✓ Generated Successfully" (2s) → hides (Finalise appears)

### Finalise Now Button
- Hidden when: no PDF generated
- Visible when: PDF exists and not stale from Stage 1 edits
- Enabled when: all fields complete (Stage 1 + Stage 2 uploads)
- Shows confirmation dialog before finalizing (with remaining days info)
- After finalizing: button shows "✓ Finalised" (greyed) → navigates back to entry list
- Entry becomes read-only after finalization

### Permanently Locked
- After second finalization (edit granted → finalised again): `permanentlyLocked = true`
- Request Edit: BLOCKED (permanently disabled)
- Request Delete: ALWAYS available regardless of lock state
- View: always available

### Delete Rules
- DRAFT and editable GENERATED: user can delete directly
- Finalized: user must request delete (admin approval → archive)
- EDIT_GRANTED: user can delete directly
- Archived: admin can permanently delete or restore

### Request System
- One request per entry at a time (edit OR delete, not both)
- Delete request auto-cancels pending edit request
- Max 3 requests per entry per month (combined edit + delete)
- Request on a Win entry → streak permanently removed

## STREAK SYSTEM (FINAL SPEC)

### Two Checkpoints Only:
1. **Generate PDF → gate to Activated** (check end date, if future = streak eligible)
2. **Finalise Entry → gate to Wins** (eligible + all mandatory fields + valid PDF + finalized)

### Activated Conditions (ALL must be true):
- Status is GENERATED (not DRAFT, not ARCHIVED)
- pdfGenerated === true (or pdfGeneratedAt exists as fallback)
- streakEligible === true (end date was future at Generate PDF time)
- NOT finalized
- streakPermanentlyRemoved !== true

### Win Conditions (ALL must be true):
- Entry is finalized (timer expired or manual Finalise Now)
- All mandatory fields complete
- Valid (non-stale) PDF exists
- streakEligible === true
- streakPermanentlyRemoved !== true

### End Date Exception:
- End date changed to past (on save) → immediately removes from Activated (recoverable: change back + regenerate)
- End date changed to future (on save) → does NOT auto-restore, must regenerate PDF

### Permanent Removal:
- Edit/Delete request on a WIN entry → permanent removal (streakPermanentlyRemoved = true)
- Does NOT apply to Activated entries — only Wins
- Archive or restore → permanent removal

### What Does NOT Affect Streaks:
- Editing any field except end date
- PDF becoming stale
- GENERATED → DRAFT revert
- Admin granting/rejecting requests
- Sending edit/delete request on Activated entries

### Dashboard Display:
- Journey funnel: [Eligible: X] → [⚡ Activated: Y] → [🏆 Wins: Z]

### Entry Fields for Streak:
```typescript
streakEligible?: boolean              // set at Generate PDF
streakPermanentlyRemoved?: boolean    // set when Win gets request or entry restored
pdfGenerated?: boolean                // set at Generate PDF
pdfGeneratedAt?: string               // ISO timestamp
pdfStale?: boolean                    // computed from pdfSourceHash comparison
permanentlyLocked?: boolean           // set after second finalization
```

## BUGS FIXED (Sessions 2 & 3)

1. **pdfGenerated never set** — API routes set pdfGeneratedAt but not pdfGenerated. Fixed with postSave.ts + hasPdfGenerated fallback in streakProgress.ts
2. **editWindowExpiresAt set to NOW** — engine.ts correctly computes it but routes bypass engine. Fixed with postSave.ts safety net
3. **pdfStale incorrectly true** — file uploads were triggering staleness. Fixed: pdfSourceHash (getHashPayload in pdfSnapshot.ts) excludes file upload fields
4. **Dashboard shows 0 streaks** — getDashboardSummary correctly calls computeCanonicalStreakSnapshot, but entry data was missing fields. Fixed with read-time normalization in getDashboardSummary
5. **API routes missing fields in response** — fdp-attended/route.ts normalizeEntry was missing editWindowExpiresAt, pdfGenerated, streakEligible. Fixed by adding them
6. **Generate endpoint missing status fields** — /api/me/entry/generate (pdfService.ts buildPdfPatch) was missing confirmationStatus and editWindowExpiresAt. Fixed.
7. **Finalized entries showing Edit/Delete instead of View/Request** — API response missing editWindowExpiresAt so client couldn't detect finalization. Fixed.
8. **Finalise Now always disabled** — canFinalise missing PDF checks, then server rejecting EDIT_GRANTED status. Fixed by allowing EDIT_GRANTED in finalizeEntry.
9. **View mode not forced for finalized entries** — effectiveViewMode override added in adapters
10. **Request Edit appearing in status banner** — removed standalone link, kept Request Action dropdown
